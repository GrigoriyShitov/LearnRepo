# Практическое задание 5

## Дано

Пример **docker-compose.yml** с двумя экземплярами приложения и сервисами *rmq, redis, postgres, zookeeper, haproxy*

## Задание

- Сделать новый **docker-compose.yml** для своего сервиса , либо “довести до ума” данный в примере до полноценного вида
- С точки зрения docker-сетей сделать так, чтоб обе реплики приложения видели только сервис *haproxy*, но не видели сам *postgres* и *zookeeper*
- На основе конфига `postgres0.yml` создаем конфиг для второй ноды, `postgres1.yml`. Запускаем, проверяем что кластер действительно работает
- Превращаем, собственно, “два экземпляра” приложения в “две реплики” (смотрим консистентность переменных и проч., все должно быть одинаковое). Делаем балансировку трафика через *haproxy* аналогично примеру для *postgres* в `haproxy.cfg`. Закрываем доступ извне к портам реплик, доступ к эндпоинтам только через `haproxy`
- \*Придумать и реализовать, как сделать так, чтоб контейнер *pg-master* ВСЕГДА был действительно мастером кластера

## Проверка результата

- Сделать любую SQL-операцию на мастер-ноде и проверить, что она также выполнилась и на реплике
- Поиграться с включением/выключением мастер-ноды, чтоб убедиться, что происходит *failover*
- Несколько раз постучаться на единую входную точку в приложение (*haproxy*) на любой эндпоинт. Убедиться, что запросы попадают на обе реплики

## Отчетность

- файл `docker-compose.yml`
- Код приложения, если он отличается от примера по умолчанию
- файл `postgres1.yml`
- файл `haproxy.cfg`
